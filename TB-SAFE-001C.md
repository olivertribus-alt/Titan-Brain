# TB-SAFE-001C — ROS 2 Safety Supervisor Integration

## Scope

TB-SAFE-001C exposes the dependency-free `SafetySupervisor` from
`core/safety_supervisor.py` as a ROS 2 control-plane adapter.  The adapter does
not own actuator motion commands; its only hardware-facing authority is the
fail-closed request for the external safety relay.

## ROS contracts

The `titan_brain_msgs` package provides:

- `SafetyHeartbeat`: sender identity, monotonically increasing sequence and
  component status code.
- `SafetyRelayStatus`: commanded relay state, auxiliary-contact feedback and
  hardware latch indication.
- `SafetySupervisorStatus`: supervisor state, active fault identifiers and the
  current closed-loop request.

All three messages are generated by `rosidl_generate_interfaces` and are
available to downstream ROS packages after a normal `colcon build`.

## Node and topics

`SafetyLoopSupervisorNode` runs a configurable timer (the reference profile is
100 Hz) and subscribes to these bounded, best-effort heartbeat topics:

| Topic | Required sender |
| --- | --- |
| `/safety/heartbeat/control_arbiter` | `control_arbiter` |
| `/safety/heartbeat/actuator_monitor` | `actuator_monitor` |
| `/safety/heartbeat/odometry` | `odometry` |

Relay feedback is consumed from `/safety/relay_status`.  The node publishes
the authoritative relay request on `/safety/relay_command` and diagnostics on
`/safety/supervisor_status`, both with reliable volatile QoS.

Heartbeat timestamps are recorded using `time.monotonic_ns()` at ingress.  ROS
header time is not used for watchdog arithmetic because simulated ROS time can
jump or reset; this preserves the monotonic-clock invariant of the core.

## Fail-closed behavior

- Missing or unhealthy heartbeats transition the supervisor to `TRIPPED` and
  publish `REQUEST_SAFETY_OPEN`.
- Invalid or unknown relay feedback immediately latches
  `HARDWARE_FAULT_LATCH`.
- A relay driver-reported latch is propagated without attempting an automatic
  release.
- Duplicate or regressing heartbeat sequence numbers are rejected as an
  unhealthy pulse.
- Shutdown handles `KeyboardInterrupt`, `rclpy.ok()` and Jazzy's
  `_rclpy_pybind11.RCLError` so a second teardown cannot mask test results.

The relay command message deliberately reports
`FEEDBACK_UNKNOWN`: it is a command, not fabricated hardware evidence.  Only
`/safety/relay_status` can supply physical feedback.
